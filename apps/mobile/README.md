# ZOE LA Mobile — Expo Shell

Generated project scaffold for an Expo app that consumes ggen-manufactured packages and talks to local Supabase Docker.


## Doctrine

- **This directory** is the Expo project scaffold (`package.json`, `app.json`, `babel.config.js`, `metro.config.js`, `tsconfig.json`, `App.tsx`, `src/lib/supabaseClient.ts`, `scripts/verify-supabase.mjs`, and this README). Every one of those files is generated by ggen from a TTL source. Edit the ontology, run `ggen sync`, the file regenerates.
- **`../../packages/*`** are the manufactured app surfaces — types, screens, navigation, forms, routes, evidence.
- **`../../supabase/migrations/*`** are the manufactured database schema and RLS.

## Quick start

```bash
# 1. Bring up local Supabase Docker (in the repo root)
cd /Users/sac/open-ontologies
supabase start

# 2. (Optional) generate types
supabase gen types typescript --local > packages/types/supabase.ts

# 3. Install Expo deps and boot the shell
cd apps/mobile
pnpm install
pnpm start          # or `pnpm web` for browser

# 4. Verify Supabase connectivity + RLS from the host
pnpm verify:supabase
```

## What the shell does

`App.tsx` mounts the ggen-generated entry screen and runs an in-app RLS probe against `push_notification`. The probe classifies the result as `rls_blocked_ok`, `rls_permissive_warn`, or `error` and shows the verdict in the header.

`scripts/verify-supabase.mjs` does the same checks plus a reachability ping and a governance-table read, all from the host process. Use it in CI.

## Known limitations

- **Node version**: Expo SDK 52's `expo-modules-core` declares `main: src/index.ts`, which Node 22.13+ and Node 24/25 can't import via ESM. Use Node 20 LTS to run `expo start` / `expo export`. The Supabase Docker stack, `verify-supabase.mjs`, and `pnpm typecheck` all work on any Node version.
- **Session storage** uses RN AsyncStorage (TODO once `@react-native-async-storage/async-storage` is added).
- **No real auth flow yet** — the shell uses anon access only.
